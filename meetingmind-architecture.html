MeetingMind - Architecture Overview
AI meeting intelligence pipeline with React frontend, Rust orchestrator, Mistral APIs, and GPU diarization/embedding service

1) Business Context

Objectives

  • Convert meeting audio into structured insights quickly.
  • Improve speaker attribution quality with acoustic matching.
  • Surface decisions, action items, and ambiguities for follow-up.

Primary Users

  • Operations and project teams running recurring meetings.
  • Facilitators who need reliable post-meeting minutes.
  • Analysts auditing speaker contributions and meeting dynamics.

Value Delivered

  • Single upload to end-to-end transcript and analysis.
  • Real-time progress updates through SSE with polling resilience.
  • Local voiceprint retrieval without external vector DB.

3) Processing Pipeline

Each stage writes to the shared job result object. This enables both live SSE consumption and polling-based recovery from the same canonical state.

4) System Architecture (Layered)

Data/Input Layer

  • Meeting audio upload from React frontend.
  • Optional attendees list supplied with job create call.

Processing Layer

  • Rust orchestrator runs phase-based pipeline.
  • Maintains per-job state (`status`, `phase`, partial outputs).
  • Streams events and exposes polling endpoint.

AI/Service Layer

  • Mistral models for transcription and structured analysis.
  • GPU service for diarization and embeddings.
  • zvec local vector store for voiceprint identification.

Output Layer

  • Speaker timeline segments and transcript.
  • Decision ledger, ambiguities, and action items.
  • Meeting dynamics metrics for post-meeting review.

5) Features

Functional

  • Audio upload and async processing jobs.
  • Live status/events with SSE.
  • Polling fallback via `/api/jobs/{id}/result`.
  • Speaker enrollment and listing endpoints.
  • Structured analytics extraction.

Non-Functional

  • Transport resilience (SSE + polling fallback).
  • Graceful degraded mode when GPU unavailable.
  • Single API surface to simplify CORS and clients.
  • Stateless frontend and in-process backend job store.

6) Deployment and Operations

Deployment Model

  • Containerized orchestrator serves API + static frontend.
  • GPU service runs as separate Python service/endpoint.
  • Mistral API consumed externally with API key.

Prerequisites

  • `MISTRAL_API_KEY` configured in runtime environment.
  • `DIARIZATION_URL` points to reachable GPU service.
  • Optional `API_KEY` to gate all `/api` routes.

Operational Workflow

  1. Deploy orchestrator and frontend bundle.
  2. Provision/verify GPU service health endpoint.
  3. Submit jobs, monitor SSE, validate `/result` recovery.
  4. Review outputs in Timeline, Ledger, and Clarifications UI.
Data-flow risk to monitor: if SSE disconnects, clients must hydrate partial state from `/result` continuously, not only on completion. This keeps transcript/segments/analysis synchronized during fallback operation.
